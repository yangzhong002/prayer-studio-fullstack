# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Prayer Studio generates three outputs from a user's situation and selected pastor styles: scripture passages (RAG-retrieved), a sermon, and a prayer (both LLM-generated). Stack: Next.js (App Router, TS) frontend, FastAPI backend, LlamaIndex over Postgres + pgvector for scripture retrieval, OpenAI-compatible LLM/embedding provider.

## Commands

Everything is wired for Docker Compose. Copy `.env.example` to `.env` first.

```bash
docker compose up --build          # full stack: db (pgvector/pg17), backend :8000, frontend :3000
docker compose up db backend       # backend-only iteration
docker compose logs -f backend
```

Backend (FastAPI + uvicorn), run locally with [uv](https://docs.astral.sh/uv/):
```bash
cd backend
uv sync                                             # create .venv from pyproject.toml + uv.lock
uv run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
uv add <package>                                    # add a new dependency
```
API docs at `http://localhost:8000/docs`. Health: `GET /health`.

Frontend (Next.js 15 / React 19):
```bash
cd frontend
npm install
npm run dev        # next dev on :3000
npm run build
npm run start
```

No test suite or linter is configured in this repo.

## Architecture

### Single generation endpoint
`POST /api/generate` ([backend/app/api/routes/generate.py](backend/app/api/routes/generate.py)) delegates to `GenerationPipeline` ([backend/app/services/pipeline.py](backend/app/services/pipeline.py)), which performs **one RAG retrieval** for scripture, then a single LLM call that produces all three outputs:

1. `scripture` — RAG-retrieved from pgvector, filtered by `source_type=scripture`, uses `top_k_scripture`
2. `sermon` — generated by the LLM based on selected pastor style(s) and user situation (no RAG)
3. `prayer` — generated by the LLM based on selected style(s) and user situation (no RAG)

Scripture chunks are passed to `build_generation_prompt` ([backend/app/services/prompt_builder.py](backend/app/services/prompt_builder.py)), then `LLMService.generate_structured_output` uses OpenAI structured outputs (strict JSON schema) to return a `GenerationResponse`. Keep this "one call, three outputs" contract intact when modifying the pipeline.

### RAG layer
[backend/app/services/rag.py](backend/app/services/rag.py) wraps LlamaIndex — used **only for scripture retrieval**:
- One `PGVectorStore` table `prayer_chunks`, `embed_dim=1536` (matches `text-embedding-3-small`). Changing the embedding model requires matching `embed_dim` and likely a reindex.
- `SentenceSplitter` with `chunk_size`/`chunk_overlap` from settings.
- Filtering is done in Python after retrieval. Increasing `top_k` is the workaround if post-filtering drops too many results.
- `build_or_update_index_for_source` indexes a single `MaterialSource` row; `reindex_all_sources` rebuilds from Postgres truth.
- DB connection for PGVectorStore is parsed from `pgvector_connection_string` with brittle string-splitting helpers — keep that env var in standard `postgresql://user:pass@host:port/db` form.

### Materials ingestion
[backend/app/api/routes/materials.py](backend/app/api/routes/materials.py) exposes:
- `GET /api/materials/sources`
- `POST /api/materials/ingest-text` (JSON, scripture only)
- `POST /api/materials/upload` (multipart; PDFs via `pypdf` through [backend/app/utils/text_extractor.py](backend/app/utils/text_extractor.py))

Each ingest writes a `MaterialSource` row (raw text + metadata) **and** immediately calls `RAGService().build_or_update_index_for_source(...)`. Postgres is the source of truth; the vector index can be rebuilt from it.

`tags` are stored as a comma-separated string on the model but exposed as `list[str]` through `MaterialSourceOut` — convert at the boundary.

### Pastor styles
`STYLE_OPTIONS` in [backend/app/services/styles.py](backend/app/services/styles.py) is the single source of truth, served by `GET /api/styles` and consumed by the frontend `TagSelector`. Styles are used as LLM prompt parameters — no RAG filtering by style.

### Config
[backend/app/core/config.py](backend/app/core/config.py) uses `pydantic-settings`, cached via `@lru_cache`. Required env vars: `DATABASE_URL`, `PGVECTOR_CONNECTION_STRING`, `OPENAI_API_KEY`. Tunables: `CHUNK_SIZE`, `CHUNK_OVERLAP`, `TOP_K_SCRIPTURE`, `BACKEND_CORS_ORIGINS`, `MAX_UPLOAD_MB`, `UPLOAD_DIR`.

`init_db()` runs on FastAPI startup ([backend/app/main.py](backend/app/main.py)) — no Alembic; schema comes from SQLAlchemy models in [backend/app/models/db_models.py](backend/app/models/db_models.py).

### Frontend
Next.js App Router. Main page [frontend/app/page.tsx](frontend/app/page.tsx) drives `GenerationForm` → `OutputPanel`; `/materials` route hosts `MaterialManager` (scripture-only ingestion). All backend calls go through [frontend/lib/api.ts](frontend/lib/api.ts) — add new endpoints there rather than fetching inline.

`NEXT_PUBLIC_API_BASE_URL` is the backend URL the browser uses. `INTERNAL_API_BASE_URL` is what the Next.js server uses for SSR inside Docker Compose (`http://backend:8000`).

## Conventions

- Don't bypass the pipeline: new generation features (reranking, moderation, eval) should plug into `GenerationPipeline`, preserving the `GenerationResponse` shape.
- Materials ingestion is scripture-only. Sermon and prayer generation rely on the LLM's parametric knowledge + style selection.
- Uploads land under `UPLOAD_DIR` (`/app/uploads` in the container, bind-mounted to `./data/uploads`).
